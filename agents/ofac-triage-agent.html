OFAC Screening Triage Agent

Takes a sanctions-screening name match (customer vs. an SDN or Consolidated Sanctions List entry) and triages it — likely false positive, requires manual review, or likely true positive — with reasoning, not just a score. Never clears or blocks a match itself; every determination routes to a BSA Officer for sign-off.

31 C.F.R. Part 501 OFAC SDN & Consolidated Sanctions Lists 10-business-day blocking report
Why this agent exists: this domain's own control-matrix analysis (Layer 2, AML/KYC) named OFAC false-positive triage as "the clearest agent use case" in the entire AML/KYC control set — real-time screening already exists as a tool category, but the analyst time it consumes reviewing fuzzy matches that are almost always false positives ("John A. Smith" vs. an SDN entry for "Jon Smith") is a language-and-judgment task, not a screening-technology gap. This agent is that recommendation, built.

What OFAC actually requires: a confirmed SDN match must be blocked and reported to OFAC within 10 business days. There is no dollar threshold and no materiality exception — sanctions liability is effectively strict liability, which is precisely why a false positive that gets escalated as if it were confirmed is nearly as costly as a true positive that gets waved through. This agent's job is narrowing that judgment call, not automating away the analyst who makes it.

Demo Scenarios

Three illustrative alerts, one per triage outcome. Names and ID numbers are fabricated for demonstration — they are not real screening data or real sanctioned individuals. Loading a scenario switches you to Tab 1 with the fields pre-filled; run it from there.

Scenario A — common name, mismatched identifiers (expect: likely false positive)
Customer "Michael J. Chen," a U.S. national born 1988, matched against an SDN entry for "Michael Chen" with a listed DOB range of 1948–1953 and no country specified. Common name, decades-off DOB.
Scenario B — exact name and country match, missing DOB (expect: requires review)
Customer "Anastasia Volkova," Russian national, DOB not on file, matched against an SDN entry for "Anastasia Volkova" (exact name and country match) with a listed DOB range the customer's own DOB can't be compared against. The gap is the missing DOB, not the match itself.
Scenario C — name, DOB, country, and ID all match (expect: likely true positive)
Customer "Dmitri Aleksandrovich Rusakov," Russian national, DOB and passport number matched against an SDN entry with the same DOB, country, and passport number listed. This is the pattern the 10-business-day blocking clock exists for.
